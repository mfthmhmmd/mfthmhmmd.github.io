Poor Boy's Linked List

Why we do this?

Sebagai pemrogram, pasti di hidup kita pernah ditantang untuk membangun linked list dari awal, entah itu ketika ujian akhir di kampus, technical interview ketika melamar kerja, atau mungkin ketika menghadap calon mertua (kurang tahu pasti alasannya kenapa, mungkin mertua si pemrogram adalah Ilmuwan Komputer). Pada saat-saat genting seperti ini, karena kita adalah pemrogram yang baik hati dan rajin menabung (tech-debt), biasanya yang terpikir pertama kali langsung gas saja untuk membuat class baru dengan nama Node yang memiliki attribute untuk menampung data dan pointer ke Node selanjutnya, lalu ada class dengan nama LinkedList yang memiliki constructor dan menerima Node dan mempunyai methods yang beroperasi pada Node tersebut.

Cara ini cukup efektif, si pemrogram mendapat nilai A pada mata kuliah struktur data, diterima bekerja di rintisan unicorn, lantas mendapat restu untuk menikahi putri si calon mertua, dan kehidupan pemrogram berlanjut. Tapi, bagaimana kalau misal ada cara lain untuk membangun linked list?

Coba kita lihat definisi linked list di halaman wikipedia1 :

. . . It is a data structure consisting of a collection of nodes which together represent a sequence.

Ok, kita tentu tahu linked list menampung sekumpulan element secara berurutan dan dapat bertambah atau berkurang secara dinamis, tapi bagaimana sebenarnya linked list direpresentasikan?

The ‘head’ of a list is its first node. The ‘tail’ of a list may refer either to the rest of the list after the head, or to the last node in the list.

mungkin kurang lebih begini wujudnya 🤔

Lupakan sebentar cara yang sudah kita tahu sebelumnya, Object Oriented dengan class dengan segala fancy constructor, generics, getter and setter. We can’t afford that, remember, this is a poor boy’s linked list. Bayangkan yang kita miliki sekarang hanya function dan data, lagipula, kadang kita memang perlu menanggalkan sebentar apa yang kita miliki dan ‘taken for granted’ agar dapat mengapresiasi hal-hal tersebut. Thanks for coming to my TED Talk.

Implementation

Kita sudah memutuskan untuk melupakan object dan class untuk sementara dan hanya diperbolehkan menggunakan function, tapi sebentar, apakah memungkinkan?

Mari kita coba buat dengan bahasa favorit 58.3% pemrogram di jagat stackoverflow, javascript2 :

const pair = (head, tail) => () // All I know I have a head and a tail
const first = (p) => () // do something with pair and return its first element
const rest = (p) => () // do something with pair and return the rest

pair adalah fungsi yang menerima dua argumen head dan tail, kita memutuskan bahwa pair disini bertugas untuk menjadi “perekat” antara head dan tail. Sedangkan first dan rest adalah fungsi yang beroperasi pada pair. first mengembalikan head dari sebuah pair, sedangkan rest mengembalikan sisanya. That’s it.

Lalu bagaimana cara “merekatkan” kedua argumen tersebut di dalam pair? berikut salah satu ide, menggunakan kontrol if statement, super keren 😎

Tapi bagaimana kalau misal :

const pair = (head, tail) => {
    return (dispatch) => {
        return dispatch(head, tail);
    }
};

pair mengembalikan closure yang menerima suatu argumen bernama dispatch, dimana dispatch nantinya dipanggil didalam closure tersebut 3. Dengan cara ini, pair seolah membuat container4 baru ketika dipanggil, jika nantinya dispatch memerlukan head atau tail, pair meneruskan kedua argumen yang tadinya disimpan di dalam container kepada dispatch, maka pair tidak perlu tahu dispatch ketika dipanggil akan melakukan apa dan seperti apa. In a sense, kita melakukan inverse disini, alih-alih pair memilih apa yang harus dilakukan, kita membiarkan si pemanggil yang menentukan apa yang akan dilakukan.

dispatch bisa jadi apapun, misal first dan rest

const first = (p) => p((h,t) => h);
const rest = (p) => p((h,t) => t);

dan sekarang kita punya linked list, tidak percaya? sama. Mari kita coba buktikan.

const foo = pair("Alice", null);
const bar = pair("Bob", foo);
const baz = pair("John", bar);

first(foo); // Alice
first(rest(baz)); //Bob
first(rest(rest(baz))) === first(foo); // true

Fun with recursion

Ok, sekarang kita punya linked list, lalu apa yang bisa dilakukan dengan linked list? kita coba mulai dengan berhitung

const count = (l) => {
    if (l === null) {
        return 0;
    } else {
        return 1 + count(rest(l));
    }
};

count(baz); // 3

Mengakses value dengan index?

const nth = (n, l) => {
    if (l === null || n < 0) {
        throw "IndexError";
    }
    if (n === 0) {
        return first(l);
    } else {
        return nth(n-1, rest(l));
    }
};

nth(1, baz); // Bob

Sampai sini menyadari sesuatu? (singly) linked list memiliki sifat rekursif, dimana akan sangat natural jika kita beroperasi pada linked list menggunakan teknik recursion. Here is another fun one :


const amap = (f, l) => l === null? l : pair(f(first(l)), amap(f, rest(l)));

const reverse = (l) => {
    const inner = (acc, l) => { 
        if(l) {
            return inner(pair(first(l), acc), rest(l));
        }
        return acc;
    }
    return inner(null, l);
};


const hi = (x) => {
    return "hi " + x + "!"
};


const revBaz = reverse(baz);
nth(0, revBaz); // Alice
nth(2, amap(hi, revBaz)); // Hi John!

Pola-pola recursion seperti diatas akan selalu muncul lagi dan lagi jika kita bekerja dengan linked list. Tentu saja kita bisa menggunakan while/for loop untuk prosedur-prosedur diatas, but that ain’t fun, right?

Yang perlu dicatat, hanya karena kita bisa memecahkan banyak masalah secara elegan dengan recursion, doesn’t mean we should always do recursion. CMIIW, beberapa bahasa pemrograman tidak mempunyai tail call optimization secara proper. Jadi harap baca aturan pakai (documentation) dengan seksama jika anda tidak ingin meledakkan call-stack mesin anda.
