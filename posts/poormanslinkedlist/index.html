Sebagai pemrogram, pasti di hidup kita pernah ditantang untuk membangun linked list dari awal, on the spot, entah itu di kampus, technical interview di kantor baru, atau mungkin ketika menghadap calon mertua (kurang tau pasti alasannya kenapa, mungkin mertua si pemrogram adalah Donald Knuth). Pada saat-saat genting seperti ini, karena kita adalah pemrogram yang baik hati dan rajin menabung (tech-debt), biasanya yang terpikir pertama kali adalah membuat class baru dengan nama Node yang memiliki property untuk menampung data dan pointer ke Node selanjutnya, lalu ada class dengan nama LinkedList yang mempunyai constructor yang menerima Node dan mempunyai method-method yang beroperasi pada Node tersebut.

Cara ini cukup efektif, si pemrogram mendapat nilai A pada matkul struktur data, diterima bekerja di rintisan unicorn, lantas mendapat restu untuk menikahi putri si calon mertua, dan kehidupan si pemrogram berlanjut. Tapi, bagaimana kalau misal ada cara lain untuk membangun linked list?

Coba kita lihat definisi linked list di halaman wikipedia :

a linked list is a linear collection of data elements whose order is not given by their physical placement in memory. Instead, each element points to the next. It is a data structure consisting of a collection of nodes which together represent a sequence.

Ok, kita tentu tahu linked list menampung banyak element secara berurutan, tapi apa sebenarnya basic building block linked list?

The ‘head’ of a list is its first node. The ‘tail’ of a list may refer either to the rest of the list after the head, or to the last node in the list.

Jadi linked list itu sebenarnya hanya struktur data berurutan yang memiliki “head” dan “tail”?

Oh ya, coba baca lagi definisi wikipedia diatas, mbah wiki tidak secara eksplisit memberi tahu bahwa linked list harus dibangun dengan cara spesifik bukan? jadi harap lupakan sebentar cara yang sudah kita tahu sebelumnya, Object Oriented dengan class dan segala fancy constructor, generic, getter dan setter, we can’t afford that, remember, this is a poor’s man linked list. Bayangkan yang kita miliki sekarang hanya function dan data.

Mari kita coba bangun dengan bahasa favorit 58.3% pemrogram di jagat stackoverflow, javascript :

const pair = (head, tail) => () // I know I have a head and a tail, so what?
const first = (p) => () // do something with p and return its first element
const rest = (p) => () // do something with p and return the rest

Pair adalah fungsi yang menerima dua argumen head dan tail, kita memutuskan bahwa pair disini bertugas untuk menjadi “perekat” antara head dan tail. Sedangkan first dan rest adalah fungsi yang beroperasi pada pair. first mengambil head suatu linked list, sedangkan rest mengembalikan sisanya. That’s it.

Lalu bagaimana implementasi pair secara konkret? berikut salah satu contoh menggunakan if statement, super keren.

Tapi bagaimana kalau misal :

const pair = (head, tail) => {
    return (dispatch) => {
        return dispatch(head, tail);
    }
};

pair mengembalikan sebuah fungsi orde tinggi anonim yang menerima suatu argumen bernama dispatch, dimana dispatch juga merupakan fungsi orde tinggi yang dipanggil didalam fungsi anonim milik pair. wow. Sabar, sebelum kembali dengan cara lama, kadang di kehidupan ini memang perlu menanggalkan sebentar apa yang kita miliki dan “taken for granted” agar bisa lebih bersyukur dan mengapresiasi apa yang kita miliki. Thank you for attending my TED Talk.

(ps : a function that returns higher-order function that takes another function and then call it, this technique is called partial application, if you aren’t already know it you should look it up, it’s pretty cool).

Dengan cara ini, yang pair tau hanya menyimpan argumen tersebut, dan jika nantinya ada fungsi dispatch yang memerlukan head atau tail, pair hanya meneruskan kedua data tersebut kepada dispatch, terserah implementasi dispatch nantinya akan seperti apa. dispatch disini bisa jadi apapun, misal first dan rest

const first = (p) => p((h,t) => h);
const rest = (p) => p((h,t) => t);


const headSayHi = (p) => {
    return p((h,t) => {
        return "Hi " + h + "!";
    });
};

Bisa saja dispatch kita sebut bersifat polimorfik, karena mempunyai banyak “wujud” tergantung implementasinya.
